Paradigmas de Interoperabilidad FHIR

FHIR soporta 4 paradigmas de interoperabilidad, ¿o quizás 8?

Los principales paradigmas son:

  • REST: Simple, plug-and-play, aprovecha HTTP, operaciones CRUD predefinidas.
  • Transacciones / Lotes: Múltiples operaciones a la vez (relacionadas o no).
  • Documentos: Colecciones de recursos enviados como un 'paquete', primer recurso 'Composition'.
  • Mensajes: Como HL7 V2 y V3, paquete de recursos, basado en eventos, asíncrono.
  • Servicios: Cualquier otro método ad-hoc para intercambiar recursos FHIR.
  • Almacenamiento: Para almacenar recursos FHIR en repositorios clínicos genéricos.
  • Suscripciones: Similar a la mensajería, permite suscribirse a tópicos y recibir notificaciones.
  • Transferencia Masiva de Datos (Bulk Data Transfer): Para grandes volúmenes de datos de pacientes (e.g., salud poblacional, ML).

Importante: Independientemente del paradigma, el contenido (los recursos FHIR) es el mismo.

Conceptos Clave

Paquetes FHIR (FHIR Bundles):

Son conjuntos de recursos, el bloque de construcción fundamental para todos los paradigmas. El Bundle.type define el tipo (transacción, mensaje, lote, documento).

Referencias dentro del Paquete:

Referencias virtuales resueltas por el servidor FHIR, útiles cuando el identificador lógico aún no se conoce.

  • Absoluta: El recurso está en un servidor específico.
  • Relativa: El recurso está en el mismo servidor.
  • Contenida: El recurso está dentro del mismo recurso (sin identidad externa).
  • Identificador: Referencia por un identificador humano.
  • Empaquetada (Bundled): El recurso de referencia está en el mismo paquete.

Operaciones Condicionales:

Permiten crear/actualizar/eliminar si no existe, utilizando una consulta de búsqueda en el encabezado 'if-none-exist'.

Resultado de la Operación (Operation Outcome):

Recurso devuelto cuando algo sale mal en una operación.

¿Cuál Paradigma es el Mejor?

No hay absolutos; es solo orientación influenciada por la preferencia y el caso de uso. No está limitado a un solo paradigma, se pueden mezclar y combinar según el caso de uso y las capacidades del sistema.

Casos de Uso Comunes:

  • Servidor de Terminología Clínica (SNOMED CT): Posiblemente REST para búsqueda.
  • Repositorio de Documentos Clínicos (sin CDA R2/PDF): Paradigma de Documentos.
  • Directorio de Proveedores con Capacidades de Búsqueda: REST para CRUD y búsqueda.
  • Acceso a Datos Oncológicos Desidentificados (10M pacientes): Transferencia Masiva de Datos FHIR.
  • Receta Electrónica: Mensajería o Transacciones.
  • Nuevo Repositorio de Datos Clínicos No-SQL: Almacenamiento FHIR.
